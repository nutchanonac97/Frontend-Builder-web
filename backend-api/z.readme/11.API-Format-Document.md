## 1. โครงสร้าง Endpoint

### 1.1 รูปแบบ URL
```
{scheme}://{domain}/{version}/api/{service}/{subservice?}/{submodule?}/{function}
```

### 1.2 องค์ประกอบ

| องค์ประกอบ | คำอธิบาย | ตัวอย่าง | บังคับ |
|-----------|----------|----------|--------|
| `scheme` | โปรโตคอล (http/https) | `https` | ✅ |
| `domain` | โดเมน API | `localhost:5211` | ✅ |
| `version` | เวอร์ชัน API | `v1`, `v2` | ✅ |
| `service` | ชื่อ Service (ลงท้ายด้วย Service) | `OwnerControlService`, `UserService`, `OrderService` | ✅ |
| `subservice` | ชื่อ SubService (ลงท้ายด้วย Service) | `SellerService`, `LavelService`, `OrderStatusService` | ✅ |
| `submodule` | โมดูลย่อย (ไม่บังคับ) | `flashsale`, `profile`, `payment` | ❌ |
| `function` | ฟังก์ชัน (action + resource) | `getProduct`, `createOrder`, `updateUserProfile` | ✅ |

### 1.3 หลักการตั้งชื่อ

#### ชื่อฟังก์ชัน (camelCase)
- **GET**: `get{Resource}`, `getList{Resources}`, `getSearch{Resources}`
- **POST**: `create{Resource}`, `register{Resource}`
- **PUT**: `update{Resource}`, `replace{Resource}`
- **PATCH**: `patch{Resource}`, `modify{Resource}`
- **DELETE**: `delete{Resource}`, `remove{Resource}`

#### ชื่อ Service (PascalCase + ลงท้ายด้วย Service)
- ✅ ถูกต้อง: `ProductService`, `UserService`, `OrderService`
- ❌ ผิด: `products`, `Product`, `productservice`, `PRODUCTSERVICE`

#### ตัวอย่าง
```
✅ GET  /v1/api/OwnerControlService/SellerService/training/getCurrent
✅ POST /v1/api/OwnerControlService/SellerService/training/register
✅ PUT  /v1/api/OwnerControlService/SellerService/shipping/updateStatus
❌ GET  /v1/api/products/flashsale/getCurrent
❌ GET  /v1/API/Product/FlashSale/get-current
```

## 2. HTTP Methods

### 2.1 ประเภท Methods และการใช้งาน

| Method | การใช้งาน | Idempotent | Safe |
|--------|-----------|------------|------|
| `GET` | ดึงข้อมูล | ✅ | ✅ |
| `POST` | สร้างข้อมูลใหม่ | ❌ | ❌ |
| `PUT` | อัพเดทข้อมูลทั้งหมด | ✅ | ❌ |
| `PATCH` | อัพเดทข้อมูลบางส่วน | ❌ | ❌ |
| `DELETE` | ลบข้อมุล | ✅ | ❌ |

### 2.2 🔄 การเรียงลำดับ Methods ในเอกสาร

**⚠️ สำคัญ**: ใน Endpoints และเอกสาร API ทุกแห่ง **ต้องเรียงลำดับ HTTP Methods ตามนี้เท่านั้น**

```
1. GET      ← ดึงข้อมูล (อ่านอย่างเดียว)
2. POST     ← สร้างข้อมูลใหม่
3. PUT      ← อัพเดทข้อมูลทั้งหมด
4. PATCH    ← อัพเดทข้อมูลบางส่วน
5. DELETE   ← ลบข้อมูล
```

#### ✅ ตัวอย่างที่ถูกต้อง:
```csharp
// ✅ ถูกต้อง - เรียงตามลำดับมาตรฐาน
group.MapGet("/api/OwnerControlService/SellerService/training/GetUsers", GetUsers);           // 1. GET
group.MapPost("/api/OwnerControlService/SellerService/training/CreateUser", CreateUser);        // 2. POST
group.MapPut("/api/OwnerControlService/SellerService/training/UpdateUser?id={id}", UpdateUser);    // 3. PUT
group.MapPatch("/api/OwnerControlService/SellerService/training/PatchUser?id={id}", PatchUser);    // 4. PATCH
group.MapDelete("/api/OwnerControlService/SellerService/training/DeleteUser?id={id}", DeleteUser);   // 5. DELETE
```

#### ❌ ตัวอย่างที่ผิด:
```csharp
// ❌ ผิด - ไม่เรียงตามลำดับมาตรฐาน
group.MapPost("/api/OwnerControlService/SellerService/training/CreateUser", CreateUser);        // POST ก่อน GET
group.MapGet("/api/OwnerControlService/SellerService/training/GetUsers", GetUsers);           
group.MapDelete("/api/OwnerControlService/SellerService/training/DeleteUser?id={id}", DeleteUser); // DELETE ก่อน PUT/PATCH
group.MapPut("/api/OwnerControlService/SellerService/training/UpdateUser?id={id}", UpdateUser);    
```

#### 📋 ในเอกสาร API Table:
```markdown
| Method | Endpoint | คำอธิบาย | Status |
|--------|----------|----------|--------|
| GET    | /api/OwnerControlService/SellerService/training/GetUsers           | ดึงรายชื่อผู้ใช้     | ✅ |
| GET    | /api/OwnerControlService/SellerService/training/GetUsersById?id={id}      | ดึงข้อมูลผู้ใช้เดี่ยว | ✅ |
| POST   | /api/OwnerControlService/SellerService/training/CreateUser           | สร้างผู้ใช้ใหม่      | ✅ |
| PUT    | /api/OwnerControlService/SellerService/training/UpdateUser?id={id}      | อัพเดทข้อมูลผู้ใช้   | ✅ |
| PATCH  | /api/OwnerControlService/SellerService/training/PatchUser?id={id}      | แก้ไขข้อมูลบางส่วน   | ✅ |
| DELETE | /api/OwnerControlService/SellerService/training/DeleteUser?id={id}      | ลบผู้ใช้           | ✅ |
```

### 2.3 เหตุผลการเรียงลำดับ
- **GET**: อ่านข้อมูลก่อน (Safe operation)
- **POST**: สร้างข้อมูลใหม่
- **PUT/PATCH**: แก้ไขข้อมูลที่มีอยู่
- **DELETE**: ลบข้อมูล (Destructive operation สุดท้าย)

## 3. รูปแบบ Request

### 3.1 Headers (บังคับ)
```http
Authorization: Bearer {access_token}
Content-Type: application/json
Accept: application/json
X-Request-ID: {unique_request_id}  // สำหรับ tracking
```

### 3.2 Query Parameters
ใช้สำหรับ:
- การกรองข้อมูล (filtering)
- การเรียงลำดับ (sorting)
- การแบ่งหน้า (pagination)

```
GET /v1/api/OwnerControlService/SellerService/training/getListUser?category=electronics&sort=price:desc&page=1&limit=20
```

### 3.3 Request Body
```json
{
  "field1": "value1",
  "field2": "value2",
  "nestedObject": {
    "subField1": "subValue1"
  }
}
```

#### Error Response
```json

    "details": {},  // รายละเอียดเพิ่มเติม (ถ้ามี)
    "field": "fieldName"  // ฟิลด์ที่ผิดพลาด (ถ้ามี)

    "version": "v1",
    "requestId": "req_123456"
  }

```

### 4.2 HTTP Status Codes

| Status Code | ความหมาย | การใช้งาน |
|-------------|----------|-----------|
| **2xx Success** |
| 200 | OK | ดำเนินการสำเร็จ |
| 201 | Created | สร้างข้อมูลสำเร็จ |
| 204 | No Content | สำเร็จแต่ไม่มีข้อมูลส่งกลับ |
| **4xx Client Error** |
| 400 | Bad Request | ข้อมูล request ไม่ถูกต้อง |
| 401 | Unauthorized | ไม่ได้ยืนยันตัวตน |
| 403 | Forbidden | ไม่มีสิทธิ์เข้าถึง |
| 404 | Not Found | ไม่พบข้อมูล |
| 409 | Conflict | ข้อมูลขัดแย้ง |
| 422 | Unprocessable Entity | ข้อมูลไม่สามารถประมวลผลได้ |
| 429 | Too Many Requests | ส่ง request มากเกินไป |
| **5xx Server Error** |
| 500 | Internal Server Error | ข้อผิดพลาดภายในเซิร์ฟเวอร์ |
| 502 | Bad Gateway | Gateway ผิดพลาด |
| 503 | Service Unavailable | Service ไม่พร้อมให้บริการ |
| 504 | Gateway Timeout | Gateway หมดเวลา |

## 5. การจัดการ Error

### 5.1 Error Codes มาตรฐาน

| Error Code | คำอธิบาย | HTTP Status |
|------------|----------|-------------|
| `VALIDATION_ERROR` | ข้อมูลไม่ผ่านการตรวจสอบ | 400 |

### 8.3 คำอธิบายไฟล์

#### 📄 Endpoints.[Feature].cs
- จัดการ endpoint mapping
- กำหนด route และ HTTP methods
- Validate request parameters
- เรียกใช้ service methods
- จัดการ response format

```csharp
public static class AnalyticsEndpoints
{
    public static void MapAnalyticsEndpoints(this IEndpointRouteBuilder app)
    {
        var group = app.MapGroup("/v1/api")
            .WithTags("Analytics Service")
            .WithOpenApi();

        // GET endpoints (เรียงก่อนเสมอ)
        group.MapGet("/OwnerControlService/SellerService/training/getPerformanceMetrics", GetPerformanceMetrics);

        // POST endpoints
        group.MapPost("/OwnerControlService/SellerService/report/create", CreateReport);

        // PUT endpoints
        group.MapPut("/OwnerControlService/SellerService/settings/update", UpdateSettings);

        // PATCH endpoints
        group.MapPatch("/OwnerControlService/SellerService/alerts/patchThreshold", PatchThreshold);

        // DELETE endpoints
        group.MapDelete("/OwnerControlService/SellerService/report/delete?id={id}", DeleteReport);
    }
}
```

### 8.5 หลักการตั้งชื่อไฟล์

| ประเภท | รูปแบบ | ตัวอย่าง |
|--------|---------|----------|
| Endpoints | `Endpoints.[SubModuleName/Feature].cs` | `Endpoints.Analytics.cs` |
| Models | `Models.[SubModuleName/Feature].cs` | `Models.Analytics.cs` |
| Service | `Service.[SubModuleName/Feature].cs` | `Service.Analytics.cs` |


### 8.6 ข้อดีของโครงสร้างนี้

1. **Modular Structure**: แยก feature เป็นโมดูลอิสระ
2. **Clear Separation**: แยก concerns ชัดเจน (Endpoints/Models/Service)
3. **Easy Navigation**: หาไฟล์ง่ายตามโครงสร้างที่เป็นระบบ
4. **Scalable**: เพิ่ม feature ใหม่ได้ง่าย
5. **Consistent**: ทุก service ใช้โครงสร้างเดียวกัน
6. **Documentation**: เอกสารอยู่ใกล้กับโค้ด

## 📝 หมายเหตุ

1. **Method Ordering**: ⚠️ **บังคับ** - ทุก Endpoint ต้องเรียงตาม GET → POST → PUT → PATCH → DELETE เท่านั้น
2. **Versioning**: ควรใช้ URL versioning (v1, v2) สำหรับ major changes
3. **Security**: ทุก API ต้องใช้ HTTPS และมีการ Authentication
4. **Rate Limiting**: ควรมีการจำกัดจำนวนการเรียก API ต่อช่วงเวลา
5. **Monitoring**: ใช้ X-Request-ID สำหรับ tracking และ debugging
6. **Documentation**: ทุก API ต้องมีเอกสารที่สมบูรณ์และอัพเดทอยู่เสมอ
7. **Consistency**: ทุก Service ต้องใช้รูปแบบเดียวกันตามเอกสารนี้
